# Release 1C Plan: Generated Bash, Zsh, and Fish Shell Integration

## Purpose

Add maintained, opt-in shell integrations that reproduce the most important `pet` workflow missing from `snip-it`: searching with the current shell command buffer as the initial query and inserting the selected snippet back into that buffer without executing it.

The integrations must be generated by `snp`, remain auditable, avoid silently changing user configuration, and depend only on the stable machine-facing contract established by Release 1B.

The preferred command surface is:

```text
snp shell init bash
snp shell init zsh
snp shell init fish
```

Release 1C completes Release 1. It must not expand into history capture, multiline creation, import diagnostics, additional variable syntax, or auto-sync.

## Prerequisites

Confirm Release 1A and Release 1B are complete.

The following Release 1B behavior must be stable before shell adapters are implemented:

- `snp select` command name and options;
- initial-query behavior;
- raw and expanded selection modes;
- success, cancellation, and operational-failure exit statuses;
- stdout payload contract;
- controlling-terminal behavior;
- multiline payload transport policy;
- library and tag constraint behavior.

Do not build shell integrations by scraping human-readable output or by invoking `snp run` or `snp clip`.

## Goals

1. Generate syntactically valid Bash, Zsh, and Fish integration code.
2. Use the current shell buffer as the initial `snp select` query.
3. Insert the selected command into the shell buffer without execution.
4. Preserve the original buffer and cursor on cancellation.
5. Surface operational errors while distinguishing them from cancellation.
6. Support raw and expanded insertion variants.
7. Avoid keybinding changes unless explicitly requested.
8. Handle quoting, shell metacharacters, Unicode, and multiline commands as data.
9. Test actual shell behavior through shell-native or pseudo-terminal integration tests.
10. Preserve all existing shell completion behavior and existing CLI semantics.

## Non-Goals

- Automatically edit `.bashrc`, `.zshrc`, Fish configuration, or other startup files.
- Automatically source generated code.
- Bind over `Ctrl-R`, `Ctrl-S`, or another common key by default.
- Execute selected commands.
- Add shell history capture.
- Add PowerShell, Nushell, Elvish, Xonsh, or other shell-buffer integrations in this release.
- Replace existing clap-generated shell completions.
- Add an external selector dependency.
- Parse or validate selected command syntax.
- Change how `snp select` searches or expands variables.

## Proposed CLI Surface

Add a new command group:

```text
snp shell init <SHELL> [OPTIONS]
```

Supported initial shells:

```text
bash
zsh
fish
```

Recommended options:

```text
--bind
--key <KEY>
--mode <raw|expanded>
--replace
--insert
--function-prefix <PREFIX>
```

The initial implementation may keep the surface narrower. The minimum required commands are:

```text
snp shell init bash
snp shell init zsh
snp shell init fish
```

The generated script should define both raw and expanded functions/widgets even when no keybindings are requested.

Recommended generated logical names:

```text
snp-select-raw
snp-select-expanded
```

Use shell-appropriate naming if hyphens are problematic, but keep names predictable and documented.

### Default binding policy

No bindings by default.

With `--bind`, install one conservative documented binding for raw selection and optionally another for expanded selection. The exact defaults should be chosen after auditing common Bash, Zsh, Fish, fzf, terminal-flow-control, and editor bindings.

Do not use `Ctrl-R` by default because it replaces native reverse history search. Do not use `Ctrl-S` without explicitly addressing terminal flow control. A multi-key sequence or Alt-based binding is likely safer.

If no broadly safe default exists, require `--key` when `--bind` is supplied.

## Core Interaction Semantics

### 1. Initial query

Read the current shell buffer and pass it to:

```text
snp select --query <buffer> --raw
```

or the equivalent expanded form.

The shell buffer must be passed as one argument without evaluation or reinterpretation.

### 2. Replace behavior

The default migration-compatible behavior should replace the entire current buffer with the selected command because the prior buffer served as the search query.

Preserve the original buffer until selection succeeds.

An explicit insert-at-cursor variant may be added if it is straightforward, but should not complicate the initial reliable release.

### 3. Cursor placement

After replacement, place the cursor at the end of the inserted command unless a stronger shell-native convention is identified and documented.

Raw placeholders should remain ordinary editable text.

### 4. Cancellation

When `snp select` returns the documented cancellation status:

- leave the buffer byte-for-byte unchanged;
- restore the original cursor position;
- print no error;
- return control to the shell editor.

### 5. Operational failure

For non-cancellation failure:

- leave the buffer unchanged;
- allow the diagnostic from `snp select` to remain visible;
- redraw the prompt/buffer correctly;
- return a failure status from the widget where shell conventions permit.

### 6. No execution

Do not call shell accept-line, execute, eval, source, or equivalent after insertion.

The user must press Enter explicitly.

## Workstream A: Shell Code Generation Architecture

### A1. Add shell command hierarchy

Integrate a new clap group without altering the existing `snp completions` command.

The help text must distinguish:

- `snp completions`: generate command-name/argument completions;
- `snp shell init`: generate interactive shell-buffer functions/widgets.

### A2. Keep templates maintainable

Store generated shell code in dedicated modules or template files according to repository conventions.

Requirements:

- avoid large unreadable inline string literals in command dispatch;
- keep shell-specific implementations separated;
- centralize shared option naming and documented function names;
- include a generated-code header stating that output is generated by `snp` and may be inspected before sourcing;
- avoid embedding version-specific absolute paths unless deliberately requested.

### A3. Stable executable resolution

Generated functions should normally invoke `snp` through `PATH` so upgrades and package-manager paths remain transparent.

If the project supports an explicit absolute-path generation option, it must be opt-in.

Avoid resolving the binary path at generation time and hard-coding a path that may become stale after Homebrew or Cargo upgrades.

### A4. Shell capability checks

Generated scripts should fail gracefully if `snp` is not found when invoked.

Do not add expensive checks during every prompt render. Check only when the widget runs.

## Workstream B: Bash Integration

### B1. Readline function

Use Bash's programmable Readline integration.

The generated function should:

1. Save `READLINE_LINE` and `READLINE_POINT`.
2. Invoke `snp select` using `READLINE_LINE` as the query.
3. Capture the payload and exit status without losing diagnostics.
4. On success, assign the selected command to `READLINE_LINE`.
5. Set `READLINE_POINT` to the correct character/byte position expected by Bash.
6. On cancellation or failure, restore the saved line and point.

### B2. Character versus byte offsets

Bash cursor positions can interact poorly with multibyte Unicode if computed using byte length in external tools.

Prefer Bash-native length semantics that match `READLINE_POINT`, and add Unicode tests.

Do not assume `${#value}` is correct without validating behavior under supported Bash versions and locales.

### B3. Binding

Use `bind -x` or the current Bash-supported mechanism for invoking a shell function that edits `READLINE_LINE`.

Generated unbound functions must be useful independently.

With explicit binding, emit a clear comment and deterministic binding command.

### B4. Command capture

Simple `$(...)` strips trailing newlines. Use the Release 1B transport selected for multiline safety.

If JSON is used, parse it without requiring `jq`. A small shell-safe encoding or a temporary-file/file-descriptor transport may be preferable.

Do not claim multiline fidelity unless tested.

## Workstream C: Zsh Integration

### C1. ZLE widget

Define ZLE widgets that operate on:

- `BUFFER`;
- `CURSOR`;
- optionally `LBUFFER`/`RBUFFER` where useful.

The widget should save buffer and cursor state before invoking `snp select`.

### C2. Query behavior

Use the full current `BUFFER` as the initial query by default, matching the replacement semantics.

A future option may use only `LBUFFER`, but it is not required for this release.

### C3. Redraw behavior

After success, cancellation, or failure, call the appropriate ZLE redisplay/reset mechanism so terminal output from the selector does not leave a visually corrupted prompt.

Do not clear user input on operational failure.

### C4. Binding

Register widgets with `zle -N`.

Do not bind keys unless requested.

Document how users can bind their own preferred sequence with `bindkey`.

### C5. Unicode and cursor semantics

Validate Zsh's cursor indexing with Unicode and multiline buffers. Use native parameter-expansion behavior rather than external byte counts.

## Workstream D: Fish Integration

### D1. Fish commandline API

Use Fish's `commandline` builtin to read and replace the current buffer.

The generated function should:

1. Capture the current buffer and cursor.
2. Invoke `snp select` with the current buffer as query.
3. Replace the buffer only on success.
4. Restore or preserve state on cancellation/failure.
5. Position the cursor predictably.

### D2. Event and redraw behavior

Use Fish-native repaint behavior after returning from the TUI.

Do not rely on Bash/Zsh terminal-control assumptions.

### D3. Binding

Use `bind` only under explicit request.

Generated functions should be installable through normal Fish configuration mechanisms without automatic file writes.

### D4. List semantics

Fish command substitution has list-splitting semantics that can corrupt multiline or whitespace-heavy data.

Use the structured transport from Release 1B and test exact reconstruction. Avoid naïve `set selected (snp select ...)` if it splits payload content.

## Workstream E: Raw and Expanded Widgets

### E1. Raw widget

The raw widget should call the raw selection mode and insert placeholders unchanged.

This is the primary pet-migration workflow because it lets users edit values with native shell facilities.

### E2. Expanded widget

The expanded widget should call the expanded selection mode. Variable prompts are handled by `snp`; the shell receives only the final resolved command.

Cancellation during variable prompting must preserve the original shell buffer.

### E3. Function naming

Use stable documented names and avoid collisions with common shell/plugin namespaces.

A prefix such as `__snp_` for private helpers and `snp_`/`snp-` for public widgets is appropriate.

Do not expose internal helper names as API unless necessary.

## Workstream F: Keybinding Strategy

### F1. No silent key takeover

The default generated output defines functions/widgets only.

README examples may show user-selected bindings.

### F2. Explicit binding generation

If `--bind` is implemented, generated output should include:

- the exact binding;
- a comment explaining it;
- separate raw/expanded choices or a documented primary mode;
- a warning when using a key known to conflict with terminal flow control or native history.

### F3. User-provided key

If `--key` is supported, validate shell-specific syntax carefully. A single abstract key string may not map cleanly across all shells.

It may be safer to support shell-native binding fragments or documented manual examples rather than over-generalizing in Release 1.

## Workstream G: Exact Text and Multiline Transport

### G1. Do not use eval

Selected commands must be treated as opaque text.

Prohibited patterns include:

- `eval "$selected"`;
- constructing a shell command that embeds selected text;
- sourcing selected content;
- re-parsing selected text through `sh -c`;
- unquoted expansion into shell syntax.

### G2. Multiline preservation

Test insertion of commands containing:

- one newline;
- multiple newlines;
- trailing newline;
- blank lines;
- indentation;
- heredoc syntax;
- embedded quotes and substitutions.

The shell adapter should reconstruct the payload through a lossless transport established by Release 1B.

Possible transports include:

- JSON with an internal parser-free decoding strategy;
- base64 with a broadly available decoder, though external dependency portability must be considered;
- a secure temporary file whose path is emitted as metadata;
- a dedicated file-descriptor protocol.

Choose the smallest robust approach. Do not depend on `jq`.

### G3. NUL bytes

Shell command buffers cannot generally represent NUL bytes. Reject or document this boundary if the storage layer can theoretically contain them.

Ordinary TOML strings should already make this unlikely, but the limitation should be explicit in tests and documentation.

## Workstream H: Tests

### H1. Generated syntax tests

Generate each shell script and validate with the shell's parser:

```bash
bash -n generated.bash
zsh -n generated.zsh
fish --no-execute generated.fish
```

Use the actual supported command for Fish syntax checking at implementation time.

### H2. Function-definition tests

Source each generated script in an isolated shell and verify expected public functions/widgets exist.

No real user startup files should be sourced.

### H3. Behavioral shell tests

Use pseudo-terminals or shell-specific testing harnesses to verify:

- current buffer becomes initial query;
- successful selection replaces the buffer;
- cursor moves to the end;
- cancellation preserves buffer and cursor;
- operational failure preserves buffer and displays diagnostics;
- raw placeholders remain unchanged;
- expanded placeholders resolve before insertion;
- no selected command executes automatically.

### H4. Special-character tests

Use fixtures containing:

- spaces;
- quotes;
- backslashes;
- pipes;
- redirects;
- semicolons;
- command substitutions;
- wildcard characters;
- Unicode;
- tabs;
- leading hyphens;
- multiline bodies.

After insertion, compare the shell buffer to the expected command as data.

### H5. Non-execution sentinel

Include a snippet that would create a sentinel file if executed.

Invoke the shell widget and assert:

- the command appears in the buffer;
- the sentinel file does not exist;
- execution occurs only if the test explicitly sends Enter in a separate controlled assertion.

### H6. Cancellation and error status tests

Mock or drive Release 1B outcomes so every shell adapter distinguishes:

- success;
- cancellation;
- operational failure.

### H7. Cross-version testing

Test against the repository's practical supported shell versions where available.

At minimum:

- current macOS Bash constraints should be considered because macOS ships an older Bash;
- current stable Zsh;
- current stable Fish;
- Linux Bash.

Do not use syntax unavailable in the oldest version the project intends to support.

## Workstream I: Documentation

### I1. README

Add a concise section showing:

```bash
eval "$(snp shell init zsh)"
```

and equivalent Bash/Fish setup, with a pointer to the full guide.

Do not recommend `eval` for selected commands; this `eval` is only for trusted generated initialization code from the installed binary. Explain that users may inspect output before sourcing.

### I2. User guide

Document:

- difference between completions and shell integration;
- raw versus expanded widgets;
- current-buffer query behavior;
- replacement and cursor semantics;
- cancellation behavior;
- manual keybinding examples;
- `--bind` behavior if implemented;
- uninstall/removal instructions;
- troubleshooting for no controlling terminal, prompt redraw, and conflicting bindings;
- security note that insertion does not imply safety or sandboxing.

### I3. Pet migration guide

Show a direct mapping from common pet shell functions to snip-it's generated integration.

Emphasize that `snp` inserts without execution and preserves normal shell history once the user presses Enter.

### I4. Generated-code comments

Generated scripts should contain minimal useful comments and a link or command pointing to local help:

```text
snp shell init <shell> --help
```

Avoid bloating every shell startup with extensive prose.

## Packaging and Completion Interaction

Homebrew and other package managers may already install clap-generated completion files.

`snp shell init` should not require packaging additional static files. It should generate integrations at runtime.

Do not conflate the naming or installation paths of:

- Bash/Zsh/Fish completions;
- interactive shell widgets.

If future package-manager formulas choose to install generated integration files, that should be separately planned and must remain opt-in at activation time.

## Acceptance Criteria

This plan is complete when:

1. `snp shell init bash`, `zsh`, and `fish` generate valid shell code.
2. Generated code defines documented raw and expanded selection functions/widgets.
3. No keybindings are installed by default.
4. The current shell buffer is passed as the initial query.
5. Successful selection replaces the buffer with the selected command.
6. The cursor is positioned predictably.
7. Cancellation preserves the original buffer and cursor with no error noise.
8. Operational failure preserves the buffer and leaves diagnostics visible.
9. Selected commands are never executed automatically.
10. Raw placeholders are preserved.
11. Expanded selection uses Release 1B semantics.
12. Quotes, metacharacters, Unicode, tabs, and multiline commands are inserted correctly according to the documented transport contract.
13. Generated scripts pass shell syntax checks.
14. Actual shell behavior is covered by integration or pseudo-terminal tests.
15. Existing clap completions and all existing `snp` commands remain unchanged.
16. Documentation includes setup, manual binding, cancellation, troubleshooting, and removal.
17. The full validation suite passes on available platforms, with any unavailable shell/platform validation explicitly documented.

## Release 1 Exit Criteria

Release 1 is complete after Release 1A, 1B, and 1C are all implemented and validated together.

A representative user must be able to:

1. Keep their current snip-it behavior unchanged.
2. Generate integration for Bash, Zsh, or Fish.
3. Source it explicitly.
4. Type part of a command or description.
5. Invoke a widget.
6. Search through the native snip-it TUI.
7. Insert the selected command into the current buffer without execution.
8. Cancel without losing existing input.
9. Choose raw or expanded variable behavior.
10. Press Enter normally so the shell records the command in its own history.

Do not begin Release 2 history-capture and multiline-creation work until this complete path is stable in real shells.

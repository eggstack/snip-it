# Halloy-Compatible Theme System

**Status:** Planning
**Target:** snip-it TUI (`src/ui/`)
**Reference:** <https://themes.halloy.chat>

## 1. Goal & Scope

Adopt Halloy's TOML theme format for snip-it's TUI. Theme files drop into `~/.config/snp/themes/` and just work. A new `e` keybinding in NOR mode opens an in-TUI theme picker (modeled on the snippet library browser — same keybindings, no `y` copy). Selecting a theme with `Enter` persists the choice; `e` again or `q` exits without saving. The picker applies the candidate theme live to the entire TUI as the user navigates, providing an instant preview.

### In-scope

- Bundling a curated set of Halloy-compatible themes with the release binary
- A `ThemeManager` that mirrors the `LibraryManager` pattern
- Halloy TOML parser that projects onto snip-it's 10-color `Theme` struct
- In-TUI theme picker reachable via `e` in NOR mode
- Persistence in `~/.config/snp/themes.toml` with `[active]` section
- Backward compatibility with the existing `SNP_THEME` env var
- Live theme preview (re-render of chrome + re-highlight of snippet list on selection change)
- INS mode inside the theme picker for filtering themes (identical UX to the snippet browser)

### Out-of-scope (deferred)

- `snp theme` CLI subcommand (v2)
- Halloy's `font_style` (bold/italic) — parsed but ignored
- Halloy's base64-binary share format (we use the TOML download format only)
- `Color::TRANSPARENT` substitution heuristics beyond a sane fallback chain
- Forward-compat for unknown Halloy fields beyond serde's default behavior

## 3. Filesystem Layout

```
~/.config/snp/
├── themes/                     NEW
│   ├── cyber-red.toml          seeded on first init (always present)
│   ├── ferra.toml
│   ├── gruvbox-dark.toml
│   ├── dracula.toml
│   ├── catppuccin-mocha.toml
│   ├── …                       (~50 bundled)
│   └── <user-custom>.toml      user can drop their own
└── themes.toml                 NEW
    [active]
    name = "catppuccin-mocha"
```

The `themes/` directory is the source of truth for *what themes are available*. `themes.toml` records only the *active* selection. Seeding is one-shot: bundled themes are written if missing, never overwritten, so user edits and additions are preserved.

## 4. Build-Time Asset Pipeline

### 4a. Input — `themes/` directory (project root)

Contains all bundled `.toml` files. These are committed to git. The Halloy `ferra.toml` is the canonical example. Filenames must match `^[a-z0-9][a-z0-9_-]*$`.

### 4b. Compressor — `scripts/build_themes.py`

Pure-Python script using only stdlib (`tomllib` for parsing, `lzma` for compression, `base64` for encoding). Reads each `themes/*.toml`, LZMA-compresses the bytes, and emits a base64 string. Invoked manually with:

```bash
python3 scripts/build_themes.py
```

Output: a single Rust source file at `src/ui/_generated_bundled_themes.rs`. Format:

```rust
// @generated by scripts/build_themes.py — do not edit.
// Last built: 2026-06-05 from 52 themes.

/// Uncompressed hardcoded fallback theme. Embedded at compile time via
/// include_str! for fast first-paint. LZMA decoder is only invoked when
/// the user-visible `themes/` directory needs seeding.
pub const DEFAULT_BUNDLED: &str = include_str!("../../themes/cyber-red.toml");

/// LZMA-compressed (XZ format) base64-encoded bundled themes.
/// Decoded only by `bundled_themes_decoded()`.
pub const BUNDLED: &[BundledTheme] = &[
    BundledTheme { name: "ferra",         payload_b64: "QAAAAP…/" },
    BundledTheme { name: "gruvbox-dark",  payload_b64: "QAAAAP…/" },
    // … 50 more, sorted alphabetically
];

pub struct BundledTheme { pub name: &'static str, pub payload_b64: &'static str }

/// Decompresses every bundled theme into `(name, toml_text)` pairs.
/// Intended for the one-time `init_themes_dir` extraction. Do not call
/// in hot paths.
pub fn bundled_themes_decoded() -> Vec<(String, String)> {
    BUNDLED.iter().map(|b| {
        let compressed = base64::decode(b.payload_b64).expect("valid base64");
        let toml_text = lzma_rs::xz_decompress(&compressed)
            .map(|v| String::from_utf8(v).expect("utf-8"))
            .expect("valid xz");
        (b.name.to_string(), toml_text)
    }).collect()
}
```

### 4c. Regeneration — `build.rs`

A build script that re-invokes `python3 scripts/build_themes.py` if any `themes/*.toml` file has a newer mtime than `src/ui/_generated_bundled_themes.rs`. Output is checked in (so `cargo build` works without Python on the build host, but a fresh Python run overwrites it).

```rust
fn main() {
    let themes_dir = Path::new("themes");
    let out_path = Path::new("src/ui/_generated_bundled_themes.rs");
    if themes_changed(themes_dir, out_path) {
        run_python("scripts/build_themes.py");
    }
    println!("cargo:rerun-if-changed=themes/");
    println!("cargo:rerun-if-changed=scripts/build_themes.py");
}
```

### 4d. Cargo.toml additions

```toml
[build-dependencies]

[dependencies]
lzma-rs = "0.3"        # pure-Rust XZ/LZMA decoder
base64 = "0.22"         # already in project; used for decode
```

`lzma-rs` is the only runtime decoder. No C dependency. The Python script is the only encoder and is build-time only.

### 4e. `.gitignore` addition

```
src/ui/_generated_bundled_themes.rs
```

Confirmed correctness: ignoring the generated file is safe because `build.rs` regenerates it. However, in CI we may want to check it in for reproducibility; see PR 1's tasks for the decision.

## 5. Theme Module — `src/ui/theme.rs` (rewrite)

Existing 107-line file becomes ~600 lines across four logical sections.

### 5a. `Theme` struct (unchanged surface)

```rust
#[derive(Clone, Copy, Debug)]

    pub primary: Color,
    pub secondary: Color,
    pub accent: Color,
    pub background: Color,
    pub text: Color,
    pub border: Color,
    pub selected_bg: Color,
    pub muted: Color,
    pub string_color: Color,
    pub escape_color: Color,
}
```

`pub(crate)` helpers `style_fg` and `style_fg_bg` unchanged.

### 5b. Halloy schema (private)

Mirrors `data::appearance::theme::Styles` from Halloy (commit `2026.7.1`). All sub-structs are `#[derive(Default)]` and have `#[serde(default)]` for forward compat. Sub-structs:

- `HalloyTheme` — top-level (general, text, buffer, buttons, formatting)
- `HalloyGeneral` — background, border, horizontal_rule, scrollbar, unread_indicator, highlight_indicator
- `HalloyText` — primary, secondary, tertiary, success, error, warning, info, debug, trace (all `TextStyle`)
- `HalloyBuffer` — background, background_text_input, background_title_bar, border, border_selected, highlight, selection, code, nickname, url, action, topic, timestamp (mixed colors and TextStyles)
- `HalloyBufferServerMessages` — join, part, quit, default, etc. (`OptionalTextStyle`)
- `HalloyButtons` — primary, secondary (`HalloyButton` each)
- `HalloyButton` — background, background_hover, background_selected, background_selected_hover, border_active (all colors)
- `HalloyFormatting` — 16 IRC colors (white, black, blue, …, lightgrey) as `Option<Color>`

Custom deserializers:

- `TextStyle`: `untagged` enum accepting either `"#RRGGBB"` (basic) or `{ color = "#RRGGBB", font_style = "bold" }` (extended). Parses `font_style` to `Option<FontStyle>` and stores it; the field is **never read** in v1 but kept so the parsed struct faithfully round-trips.
- `OptionalTextStyle`: same but `color` is `Option<String>`.
- `FontStyle` (kebab-case): `normal | bold | italic | bold-italic` (alias for italic-bold).
- `hex_to_color(&str) -> Option<Color>`: accepts `#RRGGBB` and `#RRGGBBAA` (alpha discarded). Returns `Color::Rgb(r, g, b)`.

### 5c. Field mapping (Halloy → snp Theme)

`impl HalloyTheme { fn into_snp_theme(&self) -> Theme { … } }`

| snp Theme field | Primary Halloy source | Fallback chain |
|---|---|---|
| `primary` | `text.primary.color` | `Color::White` |
| `secondary` | `text.secondary.color` | `text.tertiary.color` |
| `accent` | `text.tertiary.color` | `text.success.color` |
| `background` | `general.background` | `buffer.background` |
| `text` | `text.primary.color` | `Color::White` |
| `border` | `general.border` | `text.secondary.color` |
| `selected_bg` | `buffer.highlight` | `buffer.selection` |
| `muted` | `buffer.timestamp.color` | `text.tertiary.color` |
| `string_color` | `formatting.green` | `text.success.color` |
| `escape_color` | `formatting.magenta` | `text.error.color` |

If a primary source is `Color::TRANSPARENT` (Halloy's `Default` for many fields), the fallback chain is consulted. Tests will cover the `Color::TRANSPARENT` substitution.

### 5d. `ThemeManager`

```rust
pub struct ThemeManager {
    config_dir: PathBuf,
    themes_dir: PathBuf,
    config_path: PathBuf, // themes.toml
    config: ThemesConfig,
}

#[derive(Debug, Default, Clone, Serialize, Deserialize)]
pub struct ThemesConfig {
    #[serde(default)]
    pub active: Option<String>, // basename without .toml
}

pub struct ThemeInfo {
    pub name: String,
    pub path: PathBuf,
    pub is_bundled: bool, // true if this name is in BUNDLED
}
```

Public API:

| Method | Returns | Purpose |
|---|---|---|
| `new() -> SnipResult<Self>` | — | Resolves paths, loads `themes.toml` |
| `init_themes_dir(&mut self) -> SnipResult<()>` | — | Creates `themes/` if missing; seeds bundled themes; never overwrites existing files |
| `list_themes(&self) -> SnipResult<Vec<ThemeInfo>>` | sorted by name | Lists all `themes/*.toml` |
| `load_theme(&self, name: &str) -> SnipResult<Theme>` | — | Parses TOML, projects to `Theme` |
| `get_active_theme_name(&self) -> Option<String>` | — | Reads from in-memory `config` |
| `set_active_theme(&mut self, name: &str) -> SnipResult<()>` | — | Persists to `themes.toml` |
| `themes_dir(&self) -> &Path` | — | Exposed for display |

Path validation on `load_theme` and `set_active_theme`: rejects names with path separators, `..`, or non-`[a-z0-9_-]` characters (matching the validator pattern from `library.rs`).

### 5e. Active-theme global

```rust
use std::sync::RwLock;

static ACTIVE_THEME: LazyLock<RwLock<Theme>> = LazyLock::new(|| {
    RwLock::new(load_initial_theme())
});

pub fn get_theme() -> Theme {
    *ACTIVE_THEME.read().expect("theme lock poisoned")
}

pub fn set_active_theme(theme: Theme) {
    *ACTIVE_THEME.write().expect("theme lock poisoned") = theme;
}

fn load_initial_theme() -> Theme {
    // 1. Try ~/.config/snp/themes.toml → ThemesConfig.active
    // 2. Else SNP_THEME env var (resolved via existing resolve_theme for
    //    "dark" | "bright" | "light" | "auto" backward compat)
    // 3. Else: parse(DEFAULT_BUNDLED) — the "cyber-red" hardcoded fallback
    // 4. Else: hardcoded DARK_THEME (last-resort safety net)
}
```

The `RwLock` is uncontended in practice (one writer on theme change, many readers each frame). Read cost is ~10 ns.

### 5f. `resolve_theme` for env-var fallback (kept)

The existing `resolve_theme(&str) -> Theme` is kept but now *also* accepts a `Path` and tries to load from disk. The behavior is:

- `"dark"` / `"bright"` / `"light"` / `"auto"` → existing const-based path
- Any other value → treated as a theme filename, looked up in `themes/`, parsed, projected

This is the cleanest way to preserve backward compat without overloading the env var.

### 6a. New state variables

Added near the existing state block (line ~95):

```rust
let mut theme_picker_mode = false;
let mut theme_picker_insert_mode = false; // sub-mode: INS for filtering
let mut theme_filter = String::new();
let mut theme_input_text = String::new();
let mut theme_filtered: Vec<usize> = Vec::new(); // indices into theme_list
let mut theme_dirty = false;
let mut theme_active_idx: usize = 0;
let mut theme_picker_manager: Option<ThemeManager> = None;
let mut theme_picker_list: Vec<ThemeInfo> = Vec::new();
let mut pending_rehighlight = false;
```

### 6b. Lazy init

At the top of the loop, after the existing debounce logic:

```rust
if theme_picker_manager.is_none() {
    let mut mgr = ThemeManager::new()?;
    mgr.init_themes_dir()?;
    theme_picker_list = mgr.list_themes()?;
    let active = mgr.get_active_theme_name();
    theme_active_idx = active
        .as_ref()
        .and_then(|n| theme_picker_list.iter().position(|t| &t.name == n))
        .unwrap_or(0);
    // Seed filter result with full list
    theme_filtered = (0..theme_picker_list.len()).collect();
    theme_picker_manager = Some(mgr);
    // Apply active theme immediately
    apply_theme_from_picker();
}
```

### 6c. Filtering logic

Identical to the snippet-browser pattern: debounce 150ms, fuzzy match using the same `MATCHER` static. The filter input is `theme_filter` (committed) and `theme_input_text` (active typing). `theme_filtered` is rebuilt from `theme_picker_list` filtered by `theme_filter`.

### 6d. NOR-mode `e` binding (in the existing NOR dispatch at `mod.rs:766`)

```rust
KeyCode::Char('e') => {
    if !theme_picker_mode {
        // Ensure picker is initialized (lazy)
        ensure_theme_picker_ready();
        theme_picker_mode = true;
        // Don't reset theme_active_idx — preserve last preview position
        // (or jump to active). Re-apply active theme to prevent stale
        // preview color.
        apply_theme_from_picker();
        pending_rehighlight = true;
    } else {
        // Already in picker: 'e' means "exit without saving" (back to snippets)
        cancel_theme_picker();
    }
}
```

### 6e. Picker dispatch — new branch above NOR/INS

```rust
if theme_picker_mode {
    if theme_picker_insert_mode {
        // INS mode — identical to snippet INS: typing, backspace, esc, ↑/↓,
        // Enter. Filter updates go into theme_filter (via debounce).
        match key.code {
            KeyCode::Esc => {
                if !theme_input_text.is_empty() {
                    theme_filter = theme_input_text.clone();
                    theme_input_text.clear();
                    theme_picker_insert_mode = false;
                    theme_dirty = true;
                } else {
                    theme_picker_insert_mode = false;
                }
            }
            KeyCode::Char(c) => {
                theme_input_text.push(c);
                theme_filter = theme_input_text.clone(); // mirror snippet INS
                theme_dirty = true;
            }
            KeyCode::Backspace => {
                if !theme_input_text.is_empty() {
                    theme_input_text.pop();
                    theme_filter = theme_input_text.clone();
                    theme_dirty = true;
                }
            }
            KeyCode::Down | KeyCode::Char('j') if theme_active_idx + 1 < theme_filtered.len() => {
                theme_active_idx += 1;
                apply_theme_from_picker();
            }
            KeyCode::Up | KeyCode::Char('k') if theme_active_idx > 0 => {
                theme_active_idx -= 1;
                apply_theme_from_picker();
            }
            KeyCode::Enter => break, // commit & exit (same as NOR Enter)
            _ => {}
        }
    } else {
        // NOR mode in picker
        match key.code {
            KeyCode::Char('i') => {
                theme_picker_insert_mode = true;
                theme_input_text = theme_filter.clone();
            }
            KeyCode::Char('q') | KeyCode::Esc => cancel_theme_picker(),
            KeyCode::Char('e') => cancel_theme_picker(), // user spec: 'e' again exits
            KeyCode::Enter => commit_theme_picker(),      // save & exit
            KeyCode::Char('j') | KeyCode::Down
                if theme_active_idx + 1 < theme_filtered.len() =>
            {
                theme_active_idx += 1;
                apply_theme_from_picker();
            }
            KeyCode::Char('k') | KeyCode::Up if theme_active_idx > 0 => {
                theme_active_idx -= 1;
                apply_theme_from_picker();
            }
            KeyCode::Char('g') if is_ctrl_key(&key, 'g') => {
                theme_active_idx = 0;
                apply_theme_from_picker();
            }
            KeyCode::Char('G') => {
                theme_active_idx = theme_filtered.len().saturating_sub(1);
                apply_theme_from_picker();
            }
            // Standard gg, paging, etc. — copied from NOR dispatch
            _ => {}
        }
    }
    // Critical: skip the rest of the loop body to avoid double-handling
    continue;
}
```

### 6f. Helper functions

```rust
fn apply_theme_from_picker() {
    if let Some(mgr) = &theme_picker_manager {
        let idx = theme_filtered.get(theme_active_idx).copied().unwrap_or(0);
        if let Some(info) = theme_picker_list.get(idx)
            && let Ok(t) = mgr.load_theme(&info.name)
        {
            set_active_theme(t);
        }
    }
    pending_rehighlight = true;
}

fn commit_theme_picker() -> SnipResult<()> {
    if let Some(mgr) = &mut theme_picker_manager {
        let idx = theme_filtered.get(theme_active_idx).copied().unwrap_or(0);
        if let Some(info) = theme_picker_list.get(idx) {
            mgr.set_active_theme(info.name.clone())?;
        }
    }
    theme_picker_mode = false;
    theme_picker_insert_mode = false;
    Ok(())
}

fn cancel_theme_picker() {
    // Revert to the previously-saved theme
    if let Some(mgr) = &theme_picker_manager
        && let Some(name) = mgr.get_active_theme_name()
        && let Ok(t) = mgr.load_theme(&name)
    {
        set_active_theme(t);
    }
    theme_picker_mode = false;
    theme_picker_insert_mode = false;
    pending_rehighlight = true;
}
```

### 6g. Draw closure branching

In the existing draw closure at `mod.rs:299-494`, before rendering the list, branch on `theme_picker_mode`:

```rust
let theme = get_theme();
// … unchanged chrome (filter input, status bar)
```

A new private function `render_theme_picker(...)` produces:

- **Filter input block** (chunks[0]): "Filter" (or "Theme Filter" label), styled identically to the snippet browser, showing `theme_input_text` in INS mode or `theme_filter` in NOR mode.
- **List block** (chunks[1]): title `Themes [N] <active-name>`. Items are theme basenames (e.g. `ferra`, `gruvbox-dark`). The active theme (per `themes.toml`) is prefixed with `★ ` and rendered in `theme.accent`. The current cursor is rendered with the standard `▶ ` highlight symbol and `theme.selected_bg`.
- **Preview block** (chunks[2]): title `Preview: <theme-name>`. Renders a swatch grid using the *currently active* (live) theme's colors:
  ```
  █ primary   █ secondary  █ accent
  Background is theme.background
  > Selected uses selected_bg
  Text is theme.text, border is theme.border
  String literal: <color sample>  Escape: \<sample\>
  ```
  This is a `Paragraph` whose spans are colored with `theme.*` — so the user sees exactly what the TUI looks like. The `★ Active` line shows below if the theme equals `themes.toml`'s active.
- **Status bar** (chunks[3]): `[THEME]<INS?> | i: filter | ↑↓: preview | enter: apply | e/q: back`

### 6h. Re-highlighting on theme change

The existing pre-compute at `mod.rs:90-93` builds `highlighted_commands: Vec<Line<'static>>` once. The colors in those lines are baked in with the theme active at that time. To support live theme changes:

```rust
if pending_rehighlight {
    // Re-tokenize every command with the current theme.
    highlighted_commands = commands.iter().map(|c| highlight_command(c)).collect();
    pending_rehighlight = false;
}
```

This is O(n) of `highlight_command` calls per theme change, not per frame. For a library of ≤ 1000 snippets, the cost is well under 50ms — imperceptible after a keypress.

If profiling later shows this is too slow for very large libraries, we can throttle (re-highlight only on selection-stop, not on every keypress) using the same debounce primitive as the filter.

## 7. Theme Field Mapping — Rationale

The Halloy schema is rich (50+ color slots). snp-it's `Theme` is flat (10 slots). The mapping is the central design decision and the most likely place we'll want to iterate. Two competing concerns:

1. **Faithfulness** — a snp Theme field should map to its semantically closest Halloy field.
2. **Visual consistency** — adjacent fields (e.g. `primary` and `secondary`) should pick from *nearby* Halloy regions so they look like a coherent palette, not random colors.

Our mapping (section 5c) deliberately clusters most fields under `text.*` and `general.*` because those are the parts of a Halloy theme that the author most carefully tuned. The `formatting.*` slots are only used for the syntax-highlighting colors (`string_color`, `escape_color`) because that's the closest analog.

If a particular Halloy theme produces an ugly snp-it result, the user can either:
- Edit the .toml file directly (it's plain text, and they own the seeded copy)
- Add a per-theme override via a future `[overrides]` section in `themes.toml` (not in scope for v1)

## 10. Risk Register

1. **Mapping subjective** — some Halloy themes will look better than others. Mitigation: keep `into_snp_theme` small and well-documented so it's easy to tweak. User can edit the seeded `.toml` to taste.
2. **`Color::TRANSPARENT` from partial themes** — fallbacks tested.
3. **`lzma-rs` decoder** is pure Rust but less battle-tested than `xz2`. Mitigation: the decoder only runs once per launch (in `init_themes_dir`), not in the hot path. If it ever fails, the bundled theme write is skipped, not fatal.
4. **Build script Python dependency** — the generated file is committed (or regenerated automatically by `build.rs`). A user without Python on their host can still build from a fresh checkout as long as the generated file is present.
5. **Re-highlight cost** — bounded by library size. Mitigated by the existing `highlight_command` performance (already measured to be < 50ms for 1000 snippets).
6. **`e` key in TUI NOR mode conflicts with `snp e` CLI alias for `snp edit`** — no conflict; the TUI `e` and CLI `e` are different surfaces. Documented in `keybindings_cmd.rs`.
7. **Theme picker pre-computed highlights** — handled by `pending_rehighlight` (section 6h).
8. **Theme name normalization** — filenames are case-sensitive in the filesystem. We lowercase on comparison for "is active" lookups (consistent with `LibraryManager`).

## 11. Open Questions

None — all design decisions resolved:

- ✅ Theme dir: `~/.config/snp/themes/`
- ✅ Persistence: `themes.toml` with `[active]` section
- ✅ Bundling: 50+ themes, lzma-compressed at build, "cyber-red" hardcoded as runtime fallback
- ✅ INS mode in picker: identical to snippet view (i enters, types filter, Esc commits, ↑/↓ navigate)
- ✅ Compression: Python script using stdlib `lzma`; Rust decoder via `lzma-rs` crate
- ✅ Re-highlight: on every selection change in picker

## 12. Acceptance Criteria

A user can:

1. Install snp-it and immediately see a beautiful theme (Cyber Red)
2. Open a snippet browser, press `e`, browse 50+ themes with full live preview
3. Press `i` to filter the theme list with the same UX as the snippet browser
4. Press `Enter` to save a choice; the choice persists across launches
5. Press `e` or `q` to back out without saving
6. Drop any `themes.halloy.chat` `.toml` file into `~/.config/snp/themes/` and it appears in the picker on next launch
7. The original `SNP_THEME=dark` env var still works (backward compat)
8. All 218 existing tests pass
9. ~30 new unit tests + 4 integration tests pass
10. `cargo clippy --all-targets -- -D warnings` is clean
11. `cargo fmt --check` is clean
12. Release binary size impact: < 100 KB (50 themes × ~2 KB compressed × 1.04 base64 overhead)

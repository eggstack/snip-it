# Status Snapshot Architecture

Deep-dive reference for the status snapshot module and `snp status` command. For auto-sync internals, see [auto_sync.md](auto_sync.md).

## Overview

The status snapshot is a read-only projection of auto-sync state into a single, structured JSON or human-readable output. It aggregates pending markers, execution locks, status persistence, and sync configuration into one coherent view. The `snp status` command exposes this projection to users and scripts.

## Module Layout

**`src/status_snapshot.rs`** — Core types and snapshot capture logic:

- `capture_snapshot()` — Entry point. Reads all auto-sync artifacts, derives top-level state, collects diagnostics. Never writes or mutates any artifact.
- `derive_top_level()` — Pure function mapping (pending, execution, attempt, config) → `TopLevelSyncState`.
- `collect_diagnostics()` — Pure function generating `StatusDiagnostic` entries from a snapshot.
- `sync_configuration_state()` — Reads `SyncSettings` and maps to `SyncConfigurationState`.
- `pending_state_view()` — Reads pending marker, maps to `PendingStateView`.
- `execution_state_view()` — Inspects execution and worker lock files, maps to `ProcessStateView`.
- `attempt_state_view()` — Derives `AttemptStateView` from `AutoSyncStatus`.

**`src/commands/status_cmd.rs`** — CLI command handler:

- `run(json, sync_only)` — Calls `capture_snapshot()`, formats output as JSON or human-readable text.

## Read-Only Guarantees

`capture_snapshot()` is strictly read-only. It never:

- Writes to pending markers, lock files, or status files
- Spawns worker processes
- Modifies file permissions
- Creates or removes files

All artifact reads use `read_to_string` or `read_state_from_dir` — no side effects. Lock inspection uses `process_alive` (a `kill -0` check) which has no observable effect on the target process.

This makes `snp status` safe to run at any time, including during active sync, without interfering with auto-sync operations.

## Top-Level State Precedence

`derive_top_level()` evaluates conditions in strict priority order. The first match wins:

| Priority | Condition | Result |
|----------|-----------|--------|
| 1 | Pending marker corrupt or inaccessible | `CorruptOrInaccessible` |
| 2 | Execution lock held by live process | `LiveExecution { pid, started_at_unix_ms }` |
| 3 | Pending exists + attention required | `PendingAttentionRequired` |
| 4 | Pending exists + backoff window active | `PendingRetryBackoff { next_attempt_at_unix_ms }` |
| 5 | Pending exists (no backoff, no attention) | `PendingAwaitingScheduling` |
| 6 | Sync configured, auto-sync enabled | `ConfiguredAndCurrent` |
| 7 | Sync configured, auto-sync disabled | `ConfiguredAutoSyncDisabled` |
| 8 | Sync not configured | `NotConfigured` |

## JSON Schema Version 1

```json
{
  "schema": 1,
  "generated_at_unix_ms": 1700000000000,
  "config_root": "/Users/user/.config/snp",
  "local": {
    "libraries": 2,
    "snippets": 42,
    "primary_library": "snippets"
  },
  "sync": {
    "configuration": "Configured",
    "top_level": "ConfiguredAndCurrent"
  },
  "pending": {
    "state": {
      "Pending": {
        "generation": 1,
        "created_at_unix_ms": 1700000000000
      }
    }
  },
  "attempt": {
    "state": "Succeeded",
    "last_attempt_generation": 1,
    "last_attempt_at_unix_ms": 1700000000000,
    "last_success_at_unix_ms": 1700000000000,
    "last_failure_class": "",
    "consecutive_failures": 0,
    "next_attempt_at_unix_ms": 0,
    "attention_required": false,
    "message": "ok"
  },
  "execution": {
    "execution_lock": "Idle",
    "worker_lock": "Idle"
  },
  "diagnostics": []
}
```

**Schema evolution:** The `schema` field starts at 1. Future versions increment this field. Consumers should reject unknown schema versions rather than silently misinterpreting data.

**Enum representation:** Rust enums are serialized as `{"Variant": { ... }}` for variants with data, or `"Variant"` for unit variants. This is serde's default externally-tagged representation.

## Human Output Format

The human-readable output is generated by `status_cmd::run(json=false, sync_only=false)`:

```
Local: 2 libraries, 42 snippets; primary=snippets
Sync: pending
Pending generation: 1
Last attempt: TransientNetwork at 2025-01-15 10:30:00
Next retry: 2025-01-15 10:35:00
Action: retry eligible at 2025-01-15 10:35:00
```

**Fields emitted conditionally:**

| Line | When shown |
|------|------------|
| `Local: ...` | `sync_only = false` |
| `Sync: ...` | Always |
| `Pending generation: N` | Pending marker exists |
| `Last attempt: ...` | `last_attempt_at_unix_ms > 0` |
| `Next retry: ...` | `next_attempt_at_unix_ms > now` |
| `Action: ...` | Top-level state is `PendingAttentionRequired`, `PendingRetryBackoff`, or `CorruptOrInaccessible` |

**Sync line variants:**

| Top-level state | Output |
|-----------------|--------|
| `CorruptOrInaccessible` | `Sync: corrupt or inaccessible state` |
| `LiveExecution` | `Sync: active (pid=N)` |
| `PendingAttentionRequired` | `Sync: attention required` |
| `PendingRetryBackoff` | `Sync: pending retry` |
| `PendingAwaitingScheduling` | `Sync: pending` |
| `ConfiguredAndCurrent` | `Sync: current` |
| `ConfiguredAutoSyncDisabled` | `Sync: auto-sync disabled` |
| `NotConfigured` | `Sync: not configured` |

## Diagnostic Codes

Diagnostics are collected by `collect_diagnostics()` and sorted by severity (Info < Warning < Error), then by code alphabetically. Each diagnostic has a severity, machine-readable code, human message, and optional remediation.

| Code | Severity | Message | Remediation |
|------|----------|---------|-------------|
| `CONFIG_LOAD_FAILED` | Error | Failed to load sync configuration | Check `~/.config/snp/sync.toml` for syntax errors |
| `NOT_CONFIGURED` | Info | Sync is not configured | Run `snp register` to configure sync |
| `PENDING_CORRUPT` | Error | Pending state is corrupt: {reason} | Remove the pending marker and retry |
| `PENDING_INACCESSIBLE` | Warning | Pending state inaccessible: {reason} | — |
| `EXECUTION_LOCK_STALE` | Warning | Execution lock held by dead process {pid} | Lock will be reclaimed on next sync attempt |
| `EXECUTION_LOCK_MALFORMED` | Warning | Execution lock file is malformed | Lock will be reclaimed on next sync attempt |
| `EXECUTION_LOCK_INACCESSIBLE` | Warning | Execution lock file is inaccessible | — |
| `WORKER_LOCK_STALE` | Warning | Worker lock held by dead process {pid} | Lock will be reclaimed on next sync attempt |
| `WORKER_LOCK_MALFORMED` | Warning | Worker lock file is malformed | Lock will be reclaimed on next sync attempt |
| `WORKER_LOCK_INACCESSIBLE` | Warning | Worker lock file is inaccessible | — |
| `ATTENTION_REQUIRED` | Error | Sync requires attention: {failure_class} | Run `snp sync` or fix the underlying issue |
| `STATUS_CORRUPT` | Error | Auto-sync status file is corrupt | Remove `auto-sync-status.toml` and retry |

**Pending reason codes** (used in `PENDING_CORRUPT` and `PENDING_INACCESSIBLE`):

| Reason code | Meaning |
|-------------|---------|
| `integrity_or_parse_failure` | CRC32 mismatch or TOML parse failure |
| `deserialization_failure` | Valid TOML but schema mismatch |
| `serialization_failure` | Write-path serialization error |
| `io_error` | Filesystem error reading pending marker |
| `lock_contention` | Pending transaction lock held |

**Process state views** (used in lock diagnostics):

| View | Meaning |
|------|---------|
| `Idle` | No lock file or empty lock file |
| `Live { pid, started_at_unix_ms }` | Lock held by a live process |
| `DeadStale { pid }` | Lock held by a dead process |
| `Malformed` | Lock file exists but cannot be parsed |
| `Inaccessible` | Lock file exists but cannot be read |

## Attempt State View

`AttemptStateView` is derived from `AutoSyncStatus`:

| Condition | View |
|-----------|------|
| `last_attempt_at_unix_ms == 0` | `NeverAttempted` |
| `attention_required == true` | `AttentionRequired` |
| `last_success >= last_attempt` | `Succeeded` |
| `next_attempt > now` | `RetryScheduled` |
| `next_attempt > 0` (but ≤ now) | `Deferred` |
| Otherwise | `Succeeded` |

## Pending State View

`PendingStateView` is derived from `read_state_from_dir()`:

| Read result | View |
|-------------|------|
| `Ok(state)` | `Pending { generation, created_at_unix_ms }` |
| `NotFound` | `None` |
| `IntegrityMismatch` / `Corrupted` | `Corrupt { reason_code }` |
| `Io(_)` | `Inaccessible { reason_code }` |
| `Deserialize(_)` | `Corrupt { reason_code }` |
| `Lock(_)` | `Inaccessible { reason_code }` |
| `Serialize(_)` | `Corrupt { reason_code }` |

## Recovery Commands

The status snapshot's `Action` field suggests recovery commands:

| Top-level state | Suggested action |
|-----------------|------------------|
| `PendingAttentionRequired` | `snp sync retry` |
| `CorruptOrInaccessible` | `snp sync repair` |
| `PendingRetryBackoff` | Wait for backoff to expire, or `snp sync retry` to force |

See [commands/sync_cmd.md](commands/sync_cmd.md) for details on each recovery command.

## Test Coverage

The module has comprehensive unit tests covering:

- Top-level state derivation for all 8 states
- Attempt state view for all 5 states
- Pending state view for all read-error variants
- Process state view for live/dead/malformed/idle/empty lock files
- Diagnostic ordering (severity then code)
- JSON roundtrip serialization
- Snapshot schema version pinning

All tests use `TempDir` with explicit state directories — no global state mutation.
